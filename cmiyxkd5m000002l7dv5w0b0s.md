Google の Gemini, ChatGPT などの大規模言語モデル（LLM）の利用にかかる費用は、従来のソフトウェアとは異なり、「トークンベース」で計算されます。特に開発者 API を利用する場合、この料金モデルを理解することが、コスト管理の鍵となります。

## 1. AI モデルの料金計算とトークンとは何か？

AI モデルの料金は、主に **従量課金制 (Pay-As-You-Go)** に基づき、「トークン」という単位で計算されます。

トークンは、LLM が情報を処理するために使用する基本的な単位です。一般的に、英語では約 4 文字が 1 トークンに相当しますが、日本語などの複雑な言語では、1 トークンが処理する文字数はこれより少なくなり、結果として**同じテキスト量でもトークン数が多くなる**（コストが高くなる）傾向があります。

### 日本語のトークン具体例

| 詳細 | 日本語の例 | トークン数（推定） |
| :--- | :--- | :--- |
| 日本語の文章 | 私は新しいテクノロジーであるAIの料金体系について学習しています。 | 約 25〜30 Tokens |
| (参考) 英語の場合 | I am learning about AI pricing in Japanese. | 約 8〜10 Tokens |

上記のように、日本語は英語に比べてトークンへの分割が細かいため、ドキュメントのサイズが大きいほど、入力コストが相対的に高くなることに注意が必要です。

例文:

> **人工知能の進化は、私たちの生活を劇的に変化させています。**
> *(Jinkō chinō no shinka wa, watashitachi no seikatsu o gekiteki ni henka sasete imasu.)*
> （人工知能の進化は、私たちの生活を劇的に変化させています。）

| 詳細 | 日本語 | 英語（参考比較） |
| :--- | :--- | :--- |
| **文字数 (Characters)** | 35 文字 (句読点含む) | 51 文字 (スペース、句読点含む) |
| **トークン数 (推定)** | 約 **20〜25 トークン** | 約 **10〜12 トークン** |
| **トークン分割の傾向** | 漢字やひらがなの組み合わせが細かく分割されるため、**1.5〜2 文字で 1 トークン**程度になる傾向があります。 | **4 文字で 1 トークン**程度になる傾向があります。 |

#### 分割の様子（推定）:

実際のトークン分割はモデル内部の仕様によりますが、一般的には以下のように分割されます。

* 人工 / 知能 / の / 進化 / は / 、 / 私たち / の / 生活 / を / 劇的 / に / 変化 / させて / い / ます / 。

このように、日本語では単語ごと、あるいは助詞や助動詞（「の」「は」「てい」など）が個別のトークンとして処理されるため、同じ情報量でも英語に比べてトークン総数が多くなる傾向があります。

## 2. 2種類の料金設定：Input/1M と Output/1M

料金は、モデルへの送信データと、モデルからの応答データの 2 種類に分かれて計算されます。

-   **入力 100万トークンあたりの価格 (Input/1M):**
    -   意味: モデルに送信するデータ量（質問、要約対象の文書、コンテキスト情報など）に基づいて課金されます。
    -   計算: 100万トークン（1M）ごとの固定価格が適用されます。

-   **出力 100万トークンあたりの価格 (Output/1M):**
    -   意味: モデルが応答として返すデータ量（AI の回答、生成されたコード、翻訳結果など）に基づいて課金されます。
    -   計算: 100万トークン（1M）ごとの固定価格が適用されます。

### 費用の計算例

Gemini 1.5 Flash の料金を仮定します。入力 1M あたり 0.35 USD、出力 1M あたり 0.50 USD。

| 費用項目 | シナリオ | 費用（USD） |
| :--- | :--- | :--- |
| 入力コスト | 100,000 トークンを入力 | (100,000 / 1,000,000) x $0.35 = $0.035 |
| 出力コスト | 50,000 トークンを出力 | (50,000 / 1,000,000) x $0.50 = $0.025 |
| 合計 | | $0.060 |

## 3. 最も重要なコスト管理要素

従量課金制において、開発者が管理すべき最も重要な要素は **入力のサイズ（Input size）** です。

-   **入力数が圧倒的に多い**
    -   モデルに長いドキュメント（例：300 ページ）を送信して要約させる場合、費用の大部分（99%近く）は入力トークンから発生します。
    -   Gemini 1.5 Pro のような大規模なコンテキストウィンドウを持つモデルを利用する際は、特に注意が必要です。

-   **出力コストは単価が高い傾向**
    -   モデルの応答を「生成する」作業は、「受け付ける」作業よりも計算リソースを消費するため、通常、出力トークン単価は入力トークン単価よりも高く設定されています。

最も重要な結論として、コストを最小限に抑えるためには、モデルに送信する **コンテキスト（Context: 背景情報）** を、タスクの実行に必要な量だけに絞り込み、不要なデータを排除する「入力の最適化」が不可欠です。
